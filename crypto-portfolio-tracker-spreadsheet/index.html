Crypto portfolio tracker spreadsheet: what to track if you want real wallet P&L

Most crypto portfolio trackers look fine until you move funds between wallets, LP into DeFi, claim an airdrop, or bridge assets across chains. Then the numbers drift and your "profit" turns into fiction. A good crypto portfolio tracker spreadsheet should answer one practical question: what do I actually own, what did it cost me, and what is the wallet-level P&L now?

This page is for people searching for a crypto portfolio tracker template, wallet P&L spreadsheet, or a way to track crypto manually before paying for heavier software.

What a crypto tracker must do

A useful tracker is more than a holdings list. It needs to do these jobs well:

1. Separate wallets from transactions

If you mix wallet balances and raw trade history in one sheet, reconciliation gets ugly fast.

2. Keep transfer history neutral

Moving funds from one wallet to another should not look like new profit or new spending.

3. Preserve cost basis

Without cost basis, your wallet P&L is just current value with vibes attached.

4. Tag non-trade events

Staking rewards, airdrops, bridges, and LP deposits need explicit labels or they distort returns.

5. Reconcile current balances

Periodic balance checks catch data-entry drift before tax season turns into pain.

6. Show both wallet and portfolio views

You want one tab for total exposure and another for specific wallets or chains.

The minimum tabs your spreadsheet should have

1) Wallets

One row per wallet or exchange account.

  • Wallet label
  • Chain
  • Address or account nickname
  • Type: exchange, hot wallet, cold wallet, DeFi burner, etc.
  • Status: active, archived, watch-only

2) Transactions

This is the core log. Every meaningful movement should land here.

  • Date/time
  • Wallet
  • Asset in / amount in
  • Asset out / amount out
  • USD value at time of event
  • Fee asset / fee amount / fee USD
  • Event type
  • Counterparty or protocol
  • Tx hash / notes

3) Holdings snapshot

A current-balance tab by wallet and token. This is where you compare expected balances against reality.

4) Cost basis + realized P&L

Even if you do not build a full tax engine, you should still track basic acquisition cost and realized exits.

5) Dashboard

Your summary tab should answer, at a glance:

  • Total portfolio value
  • Total cost basis
  • Unrealized P&L
  • Realized P&L
  • Top wallet exposure
  • Stablecoin balance
  • DeFi capital currently deployed

Fields that stop wallet P&L from breaking

If you only add one improvement to your current sheet, make it these columns:

  • event_type: buy, sell, transfer_in, transfer_out, bridge, stake, unstake, reward, airdrop, LP_deposit, LP_withdraw, borrow, repay
  • linked_transfer_id: lets one wallet's outflow match another wallet's inflow
  • cost_basis_usd: the amount actually at risk from acquisition
  • fee_usd: because gas quietly changes returns
  • source_of_truth: exchange export, block explorer, manual entry, or protocol dashboard

The difference between a decent crypto P&L tracker and a broken one is usually not the dashboard. It is the event tagging.

How to handle common crypto edge cases

Bridges

Treat bridge out and bridge in as linked transfers, not a sale and rebuy, unless there was a real disposal event.

Staking rewards

Log rewards separately from principal. If you blend them together, you lose clarity on yield versus capital movement.

Airdrops

Track the receipt date, token amount, and your chosen valuation method at receipt. Even if tax treatment differs by country, operationally you still want the event recorded.

LP positions and impermanent loss

LP deposits are not simple buys. Record token amounts deposited, the pool/protocol, and later token amounts withdrawn. If you want clean performance analysis, keep a separate tab for LP entries/exits and compare the withdrawal value to a hold-only baseline.

Transfers between your own wallets

This is where most sheets die. Internal transfers should preserve cost basis and should not count as fresh capital inflow.

Common mistakes

  • Counting deposits to an exchange as new purchases
  • Ignoring fees entirely
  • Using only today's prices instead of transaction-time values for historical logs
  • Lumping staking rewards into buys
  • Tracking balances without a transaction ledger
  • Letting one sheet cover too many chains without standard event labels

A simple starting structure

If you want the minimum viable setup, build these columns first:

Transactions tab starter columns
date | wallet | chain | event_type | asset_in | qty_in | asset_out | qty_out | usd_value | fee_asset | fee_qty | fee_usd | protocol | tx_hash | linked_transfer_id | notes

Then add a dashboard with totals by wallet, chain, and token class. That gets you surprisingly far, especially if your real goal is better operational clarity rather than accountant-grade reporting on day one.

Bottom line

A crypto portfolio tracker spreadsheet works when it treats the portfolio like a system instead of a watchlist. If your current setup cannot explain wallet P&L after bridges, staking, and airdrops, it is too shallow. Start with better event types, linked transfers, and a holdings reconciliation habit.
